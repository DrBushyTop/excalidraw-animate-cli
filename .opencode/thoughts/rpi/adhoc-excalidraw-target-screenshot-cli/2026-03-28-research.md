# Research: Implement a root-level Excalidraw target screenshot CLI for LLM callers

## Research Question

We need to understand how the current codebase identifies renderable Excalidraw targets, what existing rendering or browser-automation seams already exist, and where agent-facing workflow guidance is currently defined. This will clarify the current-state constraints and reusable patterns before design or implementation work continues.

## Summary

The current repository-root CLI is organized around three commands: `inspect`, `manifest init`, and `render` (`src/cli.ts:15-38`, `README.md:24-149`). Target identification is frame-first. Scene loading preserves Excalidraw element identifiers and targeting metadata such as `frameId`, `groupIds`, `name`, and `customData`, while deleted elements are removed during load (`src/io/load-excalidraw-file.ts:46-97`). Inspection then discovers frames by `type === 'frame'`, derives membership from `frameId`, and reports groups, bounds, per-element IDs, and frame metadata in a JSON shape that the rest of the CLI also uses (`src/inspect/build-inspection.ts:14-100`).

Rendering is built around a selected-target-to-SVG path. Manifest targets are either `frame` or `canvas`, with optional `elementIds` narrowing and optional `sequence` items keyed by `elementId` or `groupId` (`src/manifest/schema.ts:3-48`). `normalizeTarget()` resolves those selectors into a concrete target plan and rejects selectors outside the selected frame (`src/render/normalize-target.ts:30-163`). `renderTargetToSvg()` then exports the selected content to SVG, applies theme handling, and runs SVG animation logic in a Bun helper (`src/render/render-target-to-svg.ts:190-218`). Browser-based capture already exists in the root codebase through `captureSvgTimeline()`, which loads rendered SVG into headless Chromium and uses `page.screenshot()` to capture PNG frames for MP4/GIF/PPTX export (`src/media/capture-svg-timeline.ts:53-116`, `src/commands/render.ts:130-177`).

The repository also contains browser-oriented reference code and Playwright tests under `reference-repo/`. Those files show UI-facing export and animation patterns, including browser SVG export, screen recording, Playwright configuration, and localStorage-seeded end-to-end tests (`reference-repo/src/export.ts:3-58`, `reference-repo/src/useLoadSvg.ts:68-171`, `reference-repo/playwright.config.ts:6-63`, `reference-repo/e2e/export-svg.spec.ts:4-130`). Agent-facing workflow guidance relevant to the current CLI surface is stored in the repository README, in this screenshot feature folder, and in the related animation CLI work-item artifacts rather than in a checked-in repository `SKILL.md` file (`README.md:24-155`, `.opencode/thoughts/rpi/adhoc-excalidraw-target-screenshot-cli/ticket.md:1-17`, `.opencode/thoughts/rpi/adhoc-excalidraw-animation-cli/2026-03-28-plan.md:379-403`, `.opencode/thoughts/rpi/adhoc-excalidraw-animation-cli/2026-03-28-structure-outline.md:88-109`).

## How It Works

### 1. Scene loading preserves target metadata

`loadExcalidrawFile()` parses a `.excalidraw` file, requires a top-level object, requires `elements` to be an array when present, and returns normalized `elements`, `appState`, and `files` (`src/io/load-excalidraw-file.ts:77-97`). Each normalized element keeps `id`, `type`, `groupIds`, `name`, and optionally `frameId` and `customData`, while deleted elements are filtered out (`src/io/load-excalidraw-file.ts:46-75`, `src/io/load-excalidraw-file.ts:90-96`).

### 2. Inspection identifies frames, members, and groups

`buildInspection()` discovers frames by selecting elements where `type === 'frame'` and sorts them by ID (`src/inspect/build-inspection.ts:93-99`). For each frame, `getFrameMembers()` includes both the frame element itself and any element whose `frameId` matches the frame ID (`src/inspect/build-inspection.ts:51-55`). `getFrameGroups()` then scans those members for `groupIds` and produces per-frame group summaries (`src/inspect/build-inspection.ts:57-71`). The resulting frame report includes `id`, `name`, numeric bounds, `elementIds`, `elementCount`, `customData`, `groups`, and per-element summaries (`src/inspect/build-inspection.ts:73-90`).

The CLI exposes this through `runInspect()`, which prints either the full JSON inspection payload or a text summary (`src/commands/inspect.ts:20-56`). The test suite shows the current inspection data against `General.excalidraw`, including frame names such as `PRBOT 1`, frame `customData`, group IDs, and grouped member IDs (`tests/inspect.spec.ts:111-156`).

### 3. Manifests describe selectable target scopes

The manifest schema currently supports two target kinds: `frame` and `canvas` (`src/manifest/schema.ts:3-48`). A frame target may carry `frameId`, optional `elementIds`, and optional `sequence`; a canvas target may carry optional `elementIds` and optional `sequence` (`src/manifest/schema.ts:27-47`). Sequence items are either `elementId`-based or `groupId`-based and both carry `order` with optional `durationMs` (`src/manifest/schema.ts:11-25`).

`createManifestFromInspection()` produces one frame target per discovered frame, using the frame name when present or the frame ID when the name is blank (`src/manifest/schema.ts:50-79`). When no frames exist, it produces a single `canvas` target named `full-canvas` (`src/manifest/schema.ts:62-67`). `loadManifestFile()` validates the manifest shape and resolves `sourceFile` relative to the manifest file location (`src/manifest/load-manifest-file.ts:73-97`).

### 4. Target normalization turns manifest selectors into a render plan

`normalizeTarget()` resolves a manifest target against a loaded scene (`src/render/normalize-target.ts:152-163`). For `canvas`, it includes all scene elements (`src/render/normalize-target.ts:30-33`). For `frame`, it first verifies that the requested `frameId` matches an actual frame element and then selects the frame plus all elements whose `frameId` points to it (`src/render/normalize-target.ts:35-44`).

If `elementIds` are present, `getNarrowedElements()` restricts the target to those IDs and throws if any selector is outside the target frame (`src/render/normalize-target.ts:46-61`). If `sequence` is present, `buildOverrideByElementId()` maps both explicit `elementId` overrides and expanded `groupId` members into per-element ordering and optional duration metadata (`src/render/normalize-target.ts:67-109`). `buildStepsFromSequence()` then groups elements by effective order and returns normalized steps used later by SVG animation (`src/render/normalize-target.ts:111-150`). The tests cover default steps, subset narrowing, group expansion, override precedence, and rejection of cross-frame selectors (`tests/normalize-target.spec.ts:20-207`).

### 5. Selected targets are exported to animated SVG

`renderTargetToSvg()` begins by normalizing the target and filtering out image elements whose file blobs are unavailable (`src/render/render-target-to-svg.ts:30-38`, `src/render/render-target-to-svg.ts:190-207`). It writes a temporary payload and helper script, then runs that helper with Bun (`src/render/render-target-to-svg.ts:161-188`, `src/render/render-target-to-svg.ts:197-217`).

Inside the generated helper, the code creates a JSDOM environment, loads Excalidraw's `restore` and `exportToSvg`, restores the scene payload, exports SVG, applies theme adjustments, runs `animateSvg()`, serializes the SVG, and returns `svgText`, `finishedMs`, `width`, and `height` (`src/render/render-target-to-svg.ts:52-158`). This is the current selected-target render seam used by the CLI (`src/render/render-target-to-svg.ts:190-218`).

### 6. Browser capture exists as an internal screenshot-based media step

`captureSvgTimeline()` constructs a minimal HTML page containing the rendered SVG and a script that pauses animations and resets time to zero (`src/media/capture-svg-timeline.ts:20-51`). It launches headless Chromium via Playwright, opens the generated file URL, reads the SVG's width and height, sets the viewport to those dimensions, advances SVG time with `setCurrentTime()`, and saves one PNG per frame using `page.screenshot()` (`src/media/capture-svg-timeline.ts:53-116`).

`runRender()` uses that capture step whenever `mp4`, `gif`, or `pptx` output is requested (`src/commands/render.ts:130-168`). The same command also writes `.svg` directly from `renderTargetToSvg()` results, and it reports generated outputs on stdout (`src/commands/render.ts:118-177`). This means the current repository-root code already contains screenshot capture behavior, but it is exposed today as an internal render pipeline stage rather than as a dedicated top-level screenshot command.

### 7. Reference browser code shows related UI and export behavior

The reference app's `useLoadSvg()` loads scene data, filters deleted and missing-image elements, exports SVG in the browser, applies theme transforms, and calls `animateSvg()` in-place (`reference-repo/src/useLoadSvg.ts:82-123`). The reference export module can save SVG to disk and can record browser playback with `navigator.mediaDevices.getDisplayMedia()` and `MediaRecorder` (`reference-repo/src/export.ts:3-58`). These reference files describe current behavior in the repository but are separate from the repository-root CLI path.

## Related Patterns

### Frame-first targeting

- Frames are discovered by `element.type === 'frame'` and frame membership is derived from saved `frameId` values (`src/inspect/build-inspection.ts:51-55`, `src/inspect/build-inspection.ts:93-99`).
- Frame targets in manifests resolve back to actual frame elements before rendering (`src/render/normalize-target.ts:35-44`).

### Stable selector surfaces for callers

- Inspection reports stable frame IDs, frame names, group IDs, and per-element IDs (`src/inspect/build-inspection.ts:73-90`).
- Manifests use `frameId`, `elementIds`, `elementId`, and `groupId` as the current selector keys (`src/manifest/schema.ts:3-48`).
- Tests exercise those selectors against real frame IDs and group IDs from `General.excalidraw` (`tests/inspect.spec.ts:115-156`, `tests/normalize-target.spec.ts:62-176`).

### Manifest bootstrap and ordering

- `manifest init` creates one target per discovered frame, or `full-canvas` when no frames exist (`src/commands/manifest-init.ts:6-37`, `src/manifest/schema.ts:58-79`).
- `render` keeps output order equal to manifest target order and prefixes filenames with zero-padded ordinals (`src/commands/render.ts:118-127`, `tests/render.spec.ts:158-184`).

### Selected-target SVG reuse for downstream formats

- One normalized target plan feeds SVG generation (`src/render/render-target-to-svg.ts:190-218`).
- The same rendered SVG then feeds Playwright frame capture for MP4, GIF, and PPTX generation (`src/commands/render.ts:130-177`, `src/media/capture-svg-timeline.ts:53-116`).

### Browser automation and reference Playwright patterns

- The repository-root media path launches Playwright Chromium directly and captures screenshots programmatically (`src/media/capture-svg-timeline.ts:64-100`).
- The reference app's Playwright configuration starts a dev server and uses Chromium by default, with Firefox and WebKit locally (`reference-repo/playwright.config.ts:21-63`).
- Reference end-to-end tests commonly seed a drawing into `localStorage`, reload the page, and interact with export UI through role-based selectors (`reference-repo/e2e/export-svg.spec.ts:5-23`).

### Current caller-guidance pattern

- The README documents the workflow as `inspect` → `manifest init` → `render` with manifest examples and rule bullets (`README.md:24-149`).
- The screenshot feature request records that screenshot workflow belongs in caller guidance for this CLI (`.opencode/thoughts/rpi/adhoc-excalidraw-target-screenshot-cli/ticket.md:5-12`).
- The related animation CLI planning artifacts document the existing inspect → manifest → render instruction pattern for agent callers (`.opencode/thoughts/rpi/adhoc-excalidraw-animation-cli/2026-03-28-plan.md:393-401`, `.opencode/thoughts/rpi/adhoc-excalidraw-animation-cli/2026-03-28-structure-outline.md:101-109`).

## Architecture Overview

At the repository root, the CLI routes `inspect`, `manifest init`, and `render` through dedicated command modules (`src/cli.ts:15-38`). The `inspect` side of the system reads a raw `.excalidraw` file, normalizes target-relevant element data, and exposes a frame/group-oriented inspection surface (`src/io/load-excalidraw-file.ts:77-97`, `src/inspect/build-inspection.ts:93-99`). The manifest side turns that inspection surface into a stable target file describing which frame or canvas scope to render and, optionally, how to narrow or sequence it (`src/manifest/schema.ts:58-79`, `src/manifest/load-manifest-file.ts:73-97`).

The render side resolves one manifest target at a time into concrete scene elements, exports that selection to animated SVG, and optionally feeds the SVG into a browser capture step for raster/video packaging (`src/render/normalize-target.ts:152-163`, `src/render/render-target-to-svg.ts:190-218`, `src/media/capture-svg-timeline.ts:53-116`, `src/commands/render.ts:118-177`). The reference app remains a separate browser-oriented implementation area with its own UI-based loading, export, and Playwright test setup (`reference-repo/src/useLoadSvg.ts:68-171`, `reference-repo/src/export.ts:3-58`, `reference-repo/playwright.config.ts:6-63`).
