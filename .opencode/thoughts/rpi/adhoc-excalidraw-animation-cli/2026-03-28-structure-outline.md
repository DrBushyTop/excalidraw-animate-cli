# Implement a root-level CLI based on reference-repo for animating individual Excalidraw files with controllable frame export, reordering, and multiple output formats Structure Outline

## Design Summary

- `reference-repo/` is read-only for this work: implementation may study, copy, or adapt patterns from it, but must not edit files in that directory.
- Build a repository-root CLI that keeps the reference SVG animation core as the center, but replaces browser-UI-driven workflows with `inspect`, `manifest init`, and `render` commands.
- Use first-class Excalidraw frame membership (`frameId` plus frame elements) as the v1 targeting model, with `full-canvas` fallback and manifest-owned target ordering.
- Keep explicit sequencing metadata in a sidecar manifest that supports frame/canvas targets, group selectors, element overrides, and normalization into one execution model.
- Render GIF and MP4 through deterministic offline playback rather than browser screen capture, and export PowerPoint by embedding rendered MP4 rather than recreating native PowerPoint animation timelines.

## Patterns To Follow

- Treat files under `reference-repo/` as reference inputs only; all new behavior must land in repository-root code and supporting root-level docs/tests.
- Reuse the scene-to-SVG seam from `reference-repo/src/useLoadSvg.ts` and the timing patching model from `reference-repo/src/animate.ts` instead of inventing a separate animation engine.
- Preserve reference-style default ordering and duration behavior from `reference-repo/src/animate.ts`, but make the manifest the primary override surface instead of editor-side ID mutation in `reference-repo/src/AnimateConfig.tsx`.
- Model inspection output around first-class frame and group discovery supported by local file evidence in `General.excalidraw` and the accepted rich JSON contract from the design discussion.
- Replace browser-only export behavior from `reference-repo/src/export.ts` with an offline render path suitable for CLI execution.

## Phase Outline

### Phase 1: CLI scaffold plus inspectable manifest workflow

- Goal: Prove the root-level CLI can load `.excalidraw` input, discover frame/group structure, and emit a manifest-friendly contract without depending on the React app shell.
- Summary: Establish the new root package/CLI entrypoint, Excalidraw file loading path, and `inspect` / `manifest init` commands. This creates the authoring surface needed before render work can be validated meaningfully.
- Why this phase exists now: The later rendering phases depend on a stable target/sequence contract; without inspection and manifest bootstrap, there is no reliable way to drive or validate frame-scoped rendering.

#### File Changes

- `package.json`: add the repository-root package manifest and CLI script surface because the root currently has no package manifest.
- `General.excalidraw`: use as the concrete local validation fixture for frame discovery behavior; no source-format changes expected.
- `reference-repo/src/vendor/loadScene.ts`: copy or adapt the non-UI scene loading pattern into the new root implementation without editing this reference file.
- `reference-repo/src/useLoadSvg.ts`: follow its scene normalization pattern when defining the new root inspection/loading modules; do not edit this file.
- `File to confirm during implementation`: root CLI entrypoint module that exposes `inspect` and `manifest init`.
- `File to confirm during implementation`: root manifest schema / normalization module for `kind: frame` and `kind: canvas` targets, rich frame JSON, group summaries, and default target ordering.
- `File to confirm during implementation`: tests or fixtures covering frame discovery, no-frame fallback, and `inspect --json` output shape.

#### Validation

- Running `inspect --json` on `General.excalidraw` returns frames with IDs, names, bounds, element counts, `customData` hints, per-element group IDs, and top-level group summaries consistent with the approved design.
- Running `manifest init` on a framed drawing creates one target per discovered frame in manifest order, while a no-frame drawing produces an explicit `full-canvas` target.
- Validation rejects cross-frame narrowing mistakes and preserves the accepted rule that frame membership comes from saved `frameId` data rather than geometry heuristics.

#### Phase Boundary

- This phase does not need to render animated outputs yet.
- This phase does not need GIF, MP4, or PowerPoint export.

### Phase 2: Frame-scoped SVG animation render pipeline

- Goal: Prove the CLI can render selected frame/canvas targets through the reference animation semantics, including manifest-driven sequencing overrides.
- Summary: Add `render` support that loads a manifest, filters targets to frame membership, normalizes sequence rules, and produces animated SVG outputs or equivalent preview artifacts from the root implementation.
- Why this phase exists now: Once inspect/manifest generation is stable, the next meaningful slice is end-to-end animation correctness before introducing heavier raster/video/PPTX export dependencies.

#### File Changes

- `reference-repo/src/animate.ts`: copy or adapt the animation patching and timing model into the new root render pipeline without editing the reference file.
- `reference-repo/src/library.ts`: follow its separation of reusable animation/export helpers when shaping the root reusable modules; do not edit this file.
- `General.excalidraw`: continue using this as the fixture for validating frame filtering and sequence override behavior; no source-format changes expected.
- `File to confirm during implementation`: render orchestration module that reads the manifest, selects frame/canvas members, and normalizes implicit vs explicit sequence items.
- `File to confirm during implementation`: SVG export module that applies theme/loading normalization and writes animated SVG output for a selected target.
- `File to confirm during implementation`: tests covering default reference ordering, `groupId` simultaneous steps, explicit `elementId` override precedence, and rejection of selectors outside the target frame.

#### Validation

- Rendering a frame target with no explicit `sequence` preserves reference-style default ordering for that frame’s members.
- Rendering a manifest with `groupId` selectors produces simultaneous grouped steps, and explicit `elementId` entries override overlapping group-derived behavior.
- Rendering only selected targets excludes non-member elements and produces a reviewable animated SVG artifact per requested target.

#### Phase Boundary

- This phase does not yet require offline raster/video encoding.
- This phase does not yet create PowerPoint files.

### Phase 3: Offline media export, PowerPoint packaging, and caller guidance

- Goal: Prove the approved output pipeline works end to end for GIF, MP4, and PPTX from the same manifest-driven render flow.
- Summary: Add headless playback/capture, `ffmpeg`-backed media encoding, PowerPoint packaging with embedded MP4, and the LLM-facing usage guidance needed to make the workflow usable beyond raw SVG output.
- Why this phase exists now: Media and PowerPoint export introduce the most environment and fidelity risk, so they should build on a validated SVG render core rather than being debugged at the same time as sequencing logic.

#### File Changes

- `reference-repo/src/export.ts`: use as the behavior being replaced conceptually; keep its timing/reset ideas but do not edit this browser-capture implementation.
- `reference-repo/README.md`: follow its user-facing explanation style when writing the new root CLI documentation; do not edit the reference README unless implementation explicitly requires cross-linking.
- `File to confirm during implementation`: headless render/capture module that plays the animated SVG deterministically for offline frame extraction.
- `File to confirm during implementation`: media export module that checks for `ffmpeg` and produces MP4 plus GIF from the rendered animation timeline.
- `File to confirm during implementation`: PowerPoint export module that creates slides with embedded MP4 output per selected target.
- `File to confirm during implementation`: root README or dedicated CLI docs for `inspect`, `manifest init`, `render`, dependency checks, and expected workflow.
- `File to confirm during implementation`: agent skill/instructions artifact teaching LLM callers how to inspect, author manifests, and render outputs.
- `File to confirm during implementation`: integration tests or scripted validations for MP4/GIF generation, missing-`ffmpeg` diagnostics, and PPTX export success.

#### Validation

- The CLI produces MP4 and GIF outputs from the same manifest/render inputs, with explicit diagnostics when required dependencies such as `ffmpeg` are unavailable.
- PowerPoint export creates a PPTX containing one slide per selected target with the rendered MP4 embedded, matching the accepted v1 export scope.
- Documentation and agent guidance are sufficient for a caller to run `inspect`, create or revise a manifest, and execute `render` without relying on the React reference app.

#### Phase Boundary

- Native PowerPoint shape-by-shape animation timelines remain out of scope.
- Future arbitrary non-frame target kinds remain out of scope beyond the manifest extensibility already established.

## Risks Or Ordering Notes

- Do not edit `reference-repo/`; if implementation needs code from there, copy or reimplement it into new root-level modules so the reference app remains untouched.
- The root implementation still needs concrete module/file placement decisions because the repository has no existing root CLI scaffold; keep new files aligned to the package/command boundaries above rather than mirroring the React app wholesale.
- Headless playback fidelity versus the interactive browser reference should be validated early in Phase 3 before optimizing secondary export paths.
- `ffmpeg` and PPTX dependency setup remains an environment risk; diagnostics and installation guidance are part of the deliverable, not follow-up polish.
