# Root-level Excalidraw Animation CLI Implementation Plan

## Overview

Build a repository-root CLI that can inspect `.excalidraw` files, generate a manifest-driven animation plan, and render frame-scoped animated outputs without modifying `reference-repo/`. The implementation should preserve the reference SVG animation behavior where it is already strong, while replacing browser-only loading and export paths with deterministic CLI-friendly workflows for SVG, MP4, GIF, and PPTX.

## Current State Analysis

Today the only implementation lives under `reference-repo/` and is browser-first. Scene loading and SVG preparation are driven by `useLoadSvg()` through URL hash state and UI lifecycle, then converted to SVG and animated in-memory (`reference-repo/src/useLoadSvg.ts:82-170`). The reusable animation seam is `animateSvg()`, which sorts exported SVG nodes by ID-embedded `animateOrder` metadata, applies type-specific SMIL patches, and returns `finishedMs` for playback/export timing (`reference-repo/src/animate.ts:623-704`). Export is still tied to browser APIs such as `fileSave`, `navigator.mediaDevices.getDisplayMedia`, and `MediaRecorder` (`reference-repo/src/export.ts:1-58`).

The repository root does not yet contain a package manifest or CLI scaffold, while `General.excalidraw` already demonstrates first-class Excalidraw frame structure and frame metadata such as `name` and `customData.slidesOrder` (`General.excalidraw:8486-8525`). Prior research and design artifacts already locked key decisions around frame-based targeting, sidecar-manifest sequencing, offline media export, and PPTX embedding rather than native PowerPoint animation.

## Desired End State

The repository root contains a documented CLI with `inspect`, `manifest init`, and `render` commands. Users can target individual frames or full-canvas output, preserve default reference-style ordering when desired, override sequencing through a manifest, and render SVG, MP4, GIF, and PPTX outputs from the same normalized execution model. Success is measurable by command-level validation against `General.excalidraw`, automated tests for target selection and sequencing rules, and deterministic export diagnostics for required dependencies.

### Key Discoveries

- `animateSvg()` is the main reusable seam and should remain the center of the render pipeline (`reference-repo/src/animate.ts:643-704`).
- `useLoadSvg()` shows the exact scene-to-SVG normalization flow that should be adapted into non-React modules (`reference-repo/src/useLoadSvg.ts:99-118`).
- Browser capture in `prepareWebmData()` cannot be reused for CLI export and must be replaced, not wrapped (`reference-repo/src/export.ts:21-58`).
- `General.excalidraw` contains real frame records and frame-linked child elements, so v1 should use saved `frameId` membership rather than geometry heuristics (`General.excalidraw:8486-8525`).
- The repository has no existing root CLI, manifest validation module, ffmpeg integration, or PPTX implementation, so Phase 1 must establish clean root-level boundaries rather than mirroring the React app.

## What We're NOT Doing

- Editing files inside `reference-repo/`.
- Recreating the browser UI, toolbar stepping UX, or React editor configuration workflow at the repository root.
- Implementing native PowerPoint shape-by-shape animation timelines in v1.
- Supporting arbitrary non-frame subset targets beyond the agreed frame/canvas-extensible manifest shape.
- Using browser screen capture as the long-term CLI export path.

## Implementation Approach

Use a vertical-slice plan that first creates a trustworthy authoring surface (`inspect` and `manifest init`), then wires that contract into SVG rendering, then layers offline media and PowerPoint packaging on top. This preserves the approved structure boundaries, keeps each checkpoint reviewable, and avoids debugging sequencing, rendering, and ffmpeg/PPTX packaging all at once.

## Phase 1: Root CLI scaffold and inspection workflow

### Overview

Create the repository-root package, load `.excalidraw` files without the React shell, and expose `inspect` plus `manifest init` as the first stable contract. This phase proves that frame/group discovery and manifest bootstrap work before any export pipeline is introduced.

### Why This Phase First

Later rendering phases depend on a stable target model and normalized manifest shape. Without an inspectable contract, there is no reliable way to validate frame selection, grouping, or sequence authoring.

### Changes Required

#### 1. Root package and CLI entrypoints

**File**: `package.json`

**Changes**: Add a root package manifest with CLI scripts, build/typecheck/test commands, and a `bin` entry for the new command surface.

```json
{
  "name": "excalidraw-animate-cli",
  "private": true,
  "type": "module",
  "bin": {
    "excalidraw-animate": "./dist/cli.js"
  },
  "scripts": {
    "build": "tsc -p tsconfig.json",
    "typecheck": "tsc -p tsconfig.json --noEmit",
    "test": "vitest run"
  }
}
```

**Why**: The repository root currently has no package or CLI scaffold, so every later phase depends on this boundary existing first.

#### 2. CLI command surface and argument routing

**File**: `src/cli.ts`

**Changes**: Add a root entrypoint that dispatches `inspect`, `manifest init`, and `render` subcommands to dedicated modules.

```ts
#!/usr/bin/env node
import { runInspect } from './commands/inspect.js'
import { runManifestInit } from './commands/manifest-init.js'
import { runRender } from './commands/render.js'

const [command, ...args] = process.argv.slice(2)

if (command === 'inspect') await runInspect(args)
else if (command === 'manifest' && args[0] === 'init') await runManifestInit(args.slice(1))
else if (command === 'render') await runRender(args)
else throw new Error(`Unknown command: ${command ?? '<none>'}`)
```

**Why**: A thin routing layer keeps the command contract explicit and prevents Phase 1 from collapsing into one oversized module.

#### 3. Excalidraw file loading and inspection model

**File**: `src/io/load-excalidraw-file.ts`

**Changes**: Parse a saved `.excalidraw` JSON file into a normalized scene shape compatible with later SVG export and target selection.

```ts
export interface LoadedScene {
  elements: ExcalidrawElement[]
  appState: Record<string, unknown>
  files: BinaryFiles
}

export async function loadExcalidrawFile(filePath: string): Promise<LoadedScene> {
  const raw = JSON.parse(await fs.readFile(filePath, 'utf8'))
  return {
    elements: raw.elements.filter((el: { isDeleted?: boolean }) => !el.isDeleted),
    appState: raw.appState ?? {},
    files: raw.files ?? {},
  }
}
```

**Why**: This replaces the browser-driven loading assumptions in `useLoadSvg()` with a CLI-safe input seam.

#### 4. Frame/group discovery and inspect JSON contract

**File**: `src/inspect/build-inspection.ts`

**Changes**: Discover frames, their members, per-element group IDs, and top-level group summaries using saved `frameId` membership.

```ts
export interface InspectFrame {
  id: string
  name?: string
  bounds: { x: number; y: number; width: number; height: number }
  elementIds: string[]
  elementCount: number
  customData?: Record<string, unknown>
  groups: { id: string; elementIds: string[] }[]
}

export function buildInspection(scene: LoadedScene) {
  const frames = scene.elements.filter((el) => el.type === 'frame')
  return { frames: frames.map(toInspectFrame), hasFrames: frames.length > 0 }
}
```

**Why**: The design decisions require a rich `inspect --json` payload suitable for both humans and LLM callers.

#### 5. Manifest schema and init generation

**File**: `src/manifest/schema.ts`

**Changes**: Define the v1 manifest model for `kind: "frame"` and `kind: "canvas"`, plus future-safe extensibility and narrowing rules.

```ts
export interface ManifestTarget {
  kind: 'frame' | 'canvas'
  name: string
  frameId?: string
  elementIds?: string[]
  sequence?: SequenceItem[]
}
```

**Why**: The manifest becomes the non-destructive source of truth for ordering and target selection.

**File**: `src/commands/manifest-init.ts`

**Changes**: Emit one target per discovered frame by default, or a single `full-canvas` target when no frames exist.

```ts
const targets = inspection.hasFrames
  ? inspection.frames.map((frame) => ({ kind: 'frame', frameId: frame.id, name: frame.name ?? frame.id }))
  : [{ kind: 'canvas', name: 'full-canvas' }]
```

**Why**: This gives the workflow a fast bootstrap path that can be revised later without manual raw-file inspection.

#### 6. Phase 1 tests and fixtures

**File**: `tests/inspect.spec.ts`

**Changes**: Add tests for rich inspect output, frame membership rules, group discovery, and no-frame fallback.

```ts
it('reports frames and groups from General.excalidraw', async () => {
  const result = await inspectFile('General.excalidraw')
  expect(result.frames[0]).toMatchObject({ name: 'PRBOT 1' })
  expect(result.frames[0].groups).toBeDefined()
})
```

**Why**: This phase needs automated proof that the manifest authoring surface is trustworthy before rendering is introduced.

### Success Criteria

#### Automated Verification

- [ ] Build passes: `pnpm build`
- [ ] Tests pass: `pnpm test -- --runInBand inspect manifest`
- [ ] Type check: `pnpm typecheck`

#### Manual Verification

- [ ] `excalidraw-animate inspect General.excalidraw --json` prints frame IDs, names, bounds, element counts, `customData`, and group summaries.
- [ ] `excalidraw-animate manifest init General.excalidraw` creates one target per discovered frame in manifest order.
- [ ] A drawing without frames produces a single `full-canvas` target instead of failing silently.

## Phase 2: Manifest-driven SVG render core

### Overview

Implement frame-scoped rendering that preserves the reference animation model while moving sequencing control into the manifest. This phase should output reviewable animated SVG artifacts and prove that target filtering plus sequence normalization behave correctly.

### Why This Phase First

Once inspection and manifest generation exist, the next lowest-risk vertical slice is SVG rendering. That validates target semantics and animation correctness before introducing offline media encoding and PPTX packaging.

### Changes Required

#### 1. Root animation module adapted from reference seam

**File**: `src/render/animate-svg.ts`

**Changes**: Copy or adapt the `animateSvg()` and `getBeginTimeList()` behavior into a root-owned module, preserving existing SMIL patching semantics but allowing manifest-derived sequence metadata to override legacy ID parsing.

```ts
const explicitOrder = normalizedSequence.orderByElementId.get(element.id)
const fallbackOrder = extractLegacyOrderFromId(element)
const sortOrder = explicitOrder ?? fallbackOrder ?? sourceIndex
```

**Why**: This keeps fidelity with the proven reference patcher while shifting control to the CLI manifest.

#### 2. Target selection and normalization pipeline

**File**: `src/render/normalize-target.ts`

**Changes**: Resolve each target into its concrete element set, enforce frame-membership validation, expand `groupId` selectors to simultaneous steps, and let explicit `elementId` rules override group-derived defaults.

```ts
if (target.kind === 'frame' && element.frameId !== target.frameId) {
  throw new Error(`Element ${element.id} is outside frame ${target.frameId}`)
}
```

**Why**: This is the main rule-enforcement layer for the accepted v1 scope boundaries.

#### 3. Scene-to-SVG export orchestration

**File**: `src/render/render-target-to-svg.ts`

**Changes**: Recreate the non-UI subset of `useLoadSvg()` by filtering non-deleted elements, dropping image elements with missing blobs, exporting to SVG, applying theme transforms, and animating the result.

```ts
const svg = await exportToSvg({ elements, files: scene.files, appState: scene.appState, exportPadding: 30 })
const themedSvg = applyThemeToSvg(svg, options.theme)
const { finishedMs } = animateSvg(themedSvg, orderedElements, options)
```

**Why**: This preserves the known scene-to-SVG seam while removing React and URL-hash dependencies.

#### 4. Render command and SVG artifact output

**File**: `src/commands/render.ts`

**Changes**: Load the manifest, iterate selected targets in manifest order, render animated SVG files, and emit metadata such as `finishedMs` for later media export.

```ts
for (const target of manifest.targets) {
  const result = await renderTargetToSvg(scene, target, options)
  await writeFile(outputDirFor(target), serializeSvg(result.svg))
}
```

**Why**: This creates an end-to-end rendering slice that can be reviewed without waiting for ffmpeg or PPTX work.

#### 5. Phase 2 tests

**File**: `tests/render-svg.spec.ts`

**Changes**: Add tests for default ordering, group-based simultaneous steps, explicit override precedence, and rejection of cross-frame selectors.

```ts
it('lets explicit element overrides win over group defaults', async () => {
  const plan = normalizeTarget(target, inspection)
  expect(plan.stepsByElement.get('rect-1')?.durationMs).toBe(300)
})
```

**Why**: The accepted design decisions are mainly about selection and ordering behavior, so this phase must lock those semantics down.

### Success Criteria

#### Automated Verification

- [ ] Build passes: `pnpm build`
- [ ] Tests pass: `pnpm test -- --runInBand render-svg`
- [ ] Type check: `pnpm typecheck`

#### Manual Verification

- [ ] Rendering a frame target with no explicit `sequence` preserves reference-style ordering for that frame’s members.
- [ ] Rendering a manifest with `groupId` plus overlapping `elementId` overrides produces the expected timing order.
- [ ] `render` outputs one animated SVG per selected target and excludes elements outside the target frame.

**⏸️ PAUSE**: Wait for human verification before Phase 3

## Phase 3: Offline media export, PowerPoint packaging, and caller guidance

### Overview

Add deterministic offline playback and packaging so the same manifest-driven render flow can produce MP4, GIF, and PPTX outputs. Finish the feature by documenting the workflow and adding LLM-oriented command guidance.

### Why This Phase First

This is intentionally last because it adds the largest environment and fidelity risk. By deferring it until after SVG correctness is proven, media debugging stays isolated from target-selection and sequencing issues.

### Changes Required

#### 1. Headless playback capture for rendered SVGs

**File**: `src/media/capture-svg-timeline.ts`

**Changes**: Open rendered SVG or an HTML playback harness in headless Chromium, advance or record the animation timeline deterministically, and export image frames or a capture stream suitable for ffmpeg input.

```ts
const browser = await chromium.launch()
const page = await browser.newPage()
await page.goto(renderHarnessUrl)
await page.waitForTimeout(finishedMs)
```

**Why**: The browser-capture approach in `reference-repo/src/export.ts` depends on interactive media APIs and cannot serve as a CLI export path.

#### 2. MP4 and GIF encoding with dependency diagnostics

**File**: `src/media/export-video.ts`

**Changes**: Check for `ffmpeg`, convert captured frames or intermediate video into MP4 and GIF, and emit actionable errors when dependencies are unavailable.

```ts
await ensureFfmpeg(ffmpegPath)
await execa(ffmpegPath, ['-i', inputPath, '-vf', 'fps=12', outputGifPath])
```

**Why**: The design intentionally accepts `ffmpeg` as a v1 dependency to keep the media pipeline practical and high-confidence.

#### 3. PPTX export with embedded MP4 per target

**File**: `src/pptx/export-pptx.ts`

**Changes**: Generate one slide per selected target and embed the corresponding MP4, rather than attempting native PowerPoint animation timelines.

```ts
const slide = pptx.addSlide()
slide.addMedia({ type: 'video', path: target.mp4Path, x: 0, y: 0, w: 10, h: 5.625 })
```

**Why**: Embedded MP4 is the accepted v1 PowerPoint scope and best preserves animation fidelity.

#### 4. Root CLI docs and agent workflow guidance

**File**: `README.md`

**Changes**: Document install requirements, command flow, manifest examples, and dependency troubleshooting.

```md
1. `excalidraw-animate inspect file.excalidraw --json`
2. `excalidraw-animate manifest init file.excalidraw > animation.json`
3. `excalidraw-animate render animation.json --format mp4 --format gif --format pptx`
```

**Why**: The CLI is intended for both humans and LLM callers, so the operational workflow must be explicit.

**File**: `.opencode/skills/excalidraw-animate-cli/SKILL.md`

**Changes**: Add agent-facing instructions for inspect → revise manifest → render usage.

```md
- Run `inspect --json` first.
- Prefer frame targets in v1.
- Use explicit `elementId` entries only when refining grouped defaults.
```

**Why**: Prior design work explicitly accepted an LLM-facing workflow artifact as part of the deliverable.

#### 5. Phase 3 integration tests

**File**: `tests/media-export.spec.ts`

**Changes**: Add integration coverage for successful MP4/GIF generation, missing-ffmpeg diagnostics, and PPTX creation.

```ts
it('fails with a clear message when ffmpeg is unavailable', async () => {
  await expect(renderWithMissingFfmpeg()).rejects.toThrow(/ffmpeg/i)
})
```

**Why**: The highest-risk deliverables in this phase are environmental, so tests must cover both success and failure diagnostics.

### Success Criteria

#### Automated Verification

- [ ] Build passes: `pnpm build`
- [ ] Tests pass: `pnpm test -- --runInBand media-export pptx`
- [ ] Type check: `pnpm typecheck`

#### Manual Verification

- [ ] `render --format mp4 --format gif` produces both outputs from the same manifest/target selection.
- [ ] `render --format pptx` creates a slide deck with one slide per selected target and embedded MP4 playback.
- [ ] Missing `ffmpeg` produces explicit installation/configuration guidance instead of a low-level process error.
- [ ] The README and agent skill are sufficient for a caller to inspect, revise a manifest, and render outputs without using the React app.

## Testing Strategy

### Unit Tests

- [ ] Inspection model normalization: verifies frame discovery, group summaries, and no-frame fallback behavior.
- [ ] Manifest validation: verifies `kind`, frame narrowing, and cross-frame selector rejection.
- [ ] Sequence normalization: verifies implicit sequence synthesis, `groupId` simultaneous steps, and explicit `elementId` override precedence.
- [ ] Render metadata extraction: verifies `finishedMs` and begin-time extraction remain stable after root adaptation.

### Integration Tests

- [ ] CLI inspect flow: verifies `inspect --json` on `General.excalidraw` returns the agreed rich contract.
- [ ] CLI manifest bootstrap: verifies `manifest init` emits one frame target per discovered frame or `full-canvas` when no frames exist.
- [ ] CLI SVG render flow: verifies selected targets render to animated SVG and exclude out-of-frame elements.
- [ ] CLI media flow: verifies MP4/GIF generation and clear diagnostics when ffmpeg is missing.
- [ ] CLI PPTX flow: verifies PPTX generation with embedded MP4 for each selected target.

## Risks and Mitigations

| Risk | Mitigation |
| --- | --- |
| Root package and CLI boundaries do not exist yet | Establish package/scripts/CLI routing in Phase 1 before any render work |
| Manifest sequencing semantics drift from reference behavior | Adapt `animateSvg()` rather than redesigning it; add ordering/override tests in Phase 2 |
| Frame membership appears inconsistent with geometry | Treat saved `frameId` as truth, surface diagnostics in inspect output, and document this explicitly |
| Headless media fidelity differs from interactive playback | Validate SVG render correctness in Phase 2 before introducing capture and compare sample outputs early in Phase 3 |
| `ffmpeg` or PPTX dependencies fail in user environments | Add dependency checks, actionable CLI diagnostics, and docs during Phase 3 |
| PowerPoint export scope expands unintentionally | Keep v1 limited to embedded MP4 slides and explicitly reject native animation work in the plan/docs |
