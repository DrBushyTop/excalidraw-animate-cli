# Research: Root-level Excalidraw Animation CLI Based on `reference-repo`

## Research Question

Implement a CLI in the repository root based on the `reference-repo/` implementation that allows taking individual Excalidraw files and creating animations from them.

The requested capability set includes exporting only specific frames from an Excalidraw file and animating them, changing the order of animations instead of relying only on original draw order, getting the elements and reordering them, creating GIF and MP4 outputs, and creating PowerPoint slides with the animated elements included.

Constraints noted in the request: do not edit the `reference-repo/` code directly; copy or reimplement the relevant functionality at the repository root.

## Summary

The current repository implementation lives entirely under `reference-repo/` and is structured as a React/Vite application rather than a root-level CLI. Its animation flow loads Excalidraw scene data, exports that scene to SVG through Excalidraw APIs, and then mutates the SVG DOM to inject SMIL-based animation timing. Playback, configuration, and export are driven from browser UI components rather than command-line entrypoints.

Animation ordering is currently controlled by metadata embedded into Excalidraw element IDs. The UI in `AnimateConfig.tsx` updates element IDs with `animateOrder` and `animateDuration` markers, and the runtime in `animate.ts` sorts SVG nodes using that parsed metadata. Export support in the current codebase covers SVG and WebM. The codebase does not contain root-level implementation outside `reference-repo/`, and there is no checked-in support for GIF, MP4, or PowerPoint generation.

## How It Works

`App.tsx` holds persisted scene data in local storage and switches between animate mode and Excalidraw mode (`reference-repo/src/App.tsx:10-99`). In animate mode, `AnimateApp.tsx` invokes `useLoadSvg(initialData, theme)` and passes the resulting SVG list into `Toolbar` and `Viewer` (`reference-repo/src/AnimateApp.tsx:15-43`).

`useLoadSvg.ts` is the main loading pipeline. It reads hash parameters, then loads data from either a shared JSON scene via `loadScene()`, a library file via `loadLibraryFromBlob()`, or the in-memory initial data (`reference-repo/src/useLoadSvg.ts:126-160`, `reference-repo/src/vendor/loadScene.ts:258-288`). Each scene is normalized to non-deleted elements, image elements are filtered to those with loaded file blobs, and the scene is converted to SVG with `exportToSvg()` (`reference-repo/src/useLoadSvg.ts:99-111`). Theme adjustments are applied with `applyThemeToSvg()`, then `animateSvg()` mutates the generated SVG to insert timing and animation nodes (`reference-repo/src/useLoadSvg.ts:19-52, 112-117`).

`animate.ts` implements the core runtime behavior. It maps top-level SVG nodes back to Excalidraw elements, extracts `animateOrder` and `animateDuration` numbers from element IDs, sorts nodes by parsed order, and patches each element with SMIL animations while advancing a running `current` time cursor (`reference-repo/src/animate.ts:623-704`). Different Excalidraw element types are patched differently: text uses text-path or opacity animation, line and arrow elements animate strokes and may add pointer motion, shape primitives split outline and fill timing, freedraw interpolates path stroke progression, and images fade in (`reference-repo/src/animate.ts:263-560`). The function returns `finishedMs`, and `useLoadSvg.ts` optionally carries that time into the next scene for sequential playback across library items (`reference-repo/src/useLoadSvg.ts:82-123`).

Playback and export are browser-driven from `Toolbar.tsx`. It derives step points from SVG `<animate begin="...">` timings via `getBeginTimeList()` in `animate.ts`, uses those values for step/reset controls, and surfaces modal-driven export actions (`reference-repo/src/Toolbar.tsx:216-239`, `reference-repo/src/animate.ts:706-738`). `export.ts` provides two export helpers: one serializes the SVG after temporarily seeking to time zero, and the other prepares WebM capture by requesting browser display media, resetting SVG playback, recording with `MediaRecorder`, then stopping after the maximum `finishedMs` across the loaded SVGs (`reference-repo/src/export.ts:3-58`).

## Related Patterns

### Animation metadata pattern
- Element order and duration are encoded in Excalidraw element IDs and parsed at runtime (`reference-repo/src/AnimateConfig.tsx:39-78`, `reference-repo/src/animate.ts:623-641`)
- Elements without explicit order default to `0`, and runtime sorting falls back to SVG node order for ties (`reference-repo/src/animate.ts:631-641`; `reference-repo/README.md:44-47`)

### Scene loading pattern
- E2E tests commonly seed scene data by writing serialized drawings to localStorage and reloading the page (`reference-repo/e2e/playback.spec.ts:5-21`, `reference-repo/e2e/export-svg.spec.ts:5-20`, `reference-repo/e2e/animate-config.spec.ts:38-60`)
- Shared JSON scenes are loaded when `#json=` is present, while `.excalidrawlib` URLs are fetched and expanded into multiple scenes through the same load/export/animate pipeline (`reference-repo/src/useLoadSvg.ts:126-157`)

### Playback and export triggering pattern
- Playback is exercised in tests via both buttons and keyboard shortcuts for pause/play, stepping, reset, and toolbar visibility (`reference-repo/e2e/playback.spec.ts:24-155`)
- Export UI is modal-based and currently exposes SVG export plus a separate WebM preparation/export sequence (`reference-repo/src/Toolbar.tsx:273-307, 401-421`; `reference-repo/e2e/export-svg.spec.ts:22-111`)

### Recording/export timing pattern
- `autoplay=no` causes generated SVGs to be set to their completed animation time (`reference-repo/src/useLoadSvg.ts:137-140, 152-155`)
- WebM preparation pauses all SVGs, resets them to zero, starts browser recording, then unpauses animations for capture (`reference-repo/src/export.ts:21-57`)

## Architecture Overview

The current architecture is browser-centric. Excalidraw scene data enters through local state, shared-link loading, or library loading; `useLoadSvg.ts` converts that data into SVG; `animate.ts` injects animation semantics into the SVG DOM; `Viewer.tsx` mounts the animated SVGs for display; and `Toolbar.tsx` operates on those same SVG DOM nodes for playback and export. Export functionality currently depends on browser capabilities such as SVG DOM timing control, `getDisplayMedia`, and `MediaRecorder`.

Within the repository as it exists today, the reference implementation is the only implementation. No root-level CLI modules, no root-level rendering pipeline, and no checked-in PowerPoint export path were found outside `reference-repo/`.
